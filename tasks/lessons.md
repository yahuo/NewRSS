# Lessons

## 2026-03-27

- 当功能需要部署到其他服务器时，不能把核心链路建立在当前机器的个人脚本、skills 目录或本地命令上。先检查运行时依赖是否随仓库一起交付；如果不能，就应当内化到项目里或明确做成可安装依赖。
- 当用户给出外部工具只是作为“参考实现”时，不要把这些工具名、命令名或环境变量直接暴露到 README、接口文案和配置里。先抽象成项目自己的能力，再决定真正需要公开的运行时依赖。

## 2026-04-03

- 当用户已经提供了容器日志和运行时错误时，先以运行时证据为准，再去看本地配置和静态代码。不要因为本地环境里缺少某个变量，就过早把问题归因为“没配 API key”；先确认错误发生在抓取、判定、翻译调用还是落库阶段。

## 2026-04-10

- 给刷新链路接入外部翻译这类高成本能力时，必须先检查已有落库结果是否可复用。对同一条 entry 的重复刷新，如果正文和标题未变化，就应该复用现有翻译而不是再次调用模型；同时要避免在内容变化时错误复用旧译文。
- 给现有“create-or-update”接口新增可选字段时，缺省值不能偷偷覆盖已有状态。请求里没传这个字段时，应保留数据库中的旧值；只有显式传了 true/false 才更新。
- 当用户已经给出容器里的实际环境变量和运行日志时，不能只依据我当前看到的本地实例状态下结论。先核对“是否同一个运行实例”，再以用户提供的运行时证据为准继续定位。
- 排查翻译超时时，不能只看配置上的 `chunkMaxWords` 和并发数；必须实际检查分块结果。像 `html-chunker` 这种只按顶层节点切块的实现，遇到单个超大容器节点时会让 `chunkMaxWords` 失效，导致单块请求超时。
